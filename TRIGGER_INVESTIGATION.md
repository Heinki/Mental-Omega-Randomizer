# In-Mission Trigger Investigation

## Current Finding

The launched campaign missions such as `AREDDAWN.MAP`, `ABOTTLE.MAP`, and `EPEACE.MAP` are not present as loose files in this game folder. They are loaded from MIX archives.

After a spawned mission launch, `spawnmap.ini` is present in the game root and contains full scenario data, including `[Basic]`, `[Actions]`, `[Events]`, `[Tags]`, and `[Triggers]`. In the latest crash snapshot it contained:

- `[Basic] Name=Soviets: Ravages of War`
- `[Actions]` entries with objective text/EVA actions, for example `MISSION:LeaderOne...`
- win-related trigger records such as `COOPWIN03`, `COOPWIN04`, and `COOPWIN05`
- objective-complete UI/EVA actions such as `ObjectiveComplete` and `EVA_ObjectiveComplete`

That means the game or launch path is producing a readable merged map file. It does not yet prove that we can observe completed triggers after the mission exits, or that edits made to `spawnmap.ini` before launch will be loaded instead of overwritten.

`INI\BattleClient.ini` also contains `LongDescription` briefing text with objective lines for many campaign missions. The launcher now uses those objective lines to decide how many rewards a mission has. Missions that only say "Follow the objectives given during the mission" still need map trigger analysis to get their true objective count.

Latest passive test result: completing an objective in `ARELE.MAP` did not update `spawnmap.ini`, and the debug/syringe logs did not contain a fired objective ID. The launcher therefore cannot auto-grant objective rewards by passively watching existing files.

New active hook attempt: the launcher can extract a campaign map from `expandmo*.mix`, write a generated loose copy into the game root for the launch, patch objective-complete and winner trigger actions, and watch `debug\debug.log` for marker team launches. The generated maps are marked with:

```ini
; Generated by Mental Omega Randomizer Launcher objective hook
```

For `ABADAPPLE.MAP`, the analyzer found objective-complete action IDs `01000016`, `01000052`, `01000348`, `01000452`, plus winner action `01000352`. A dry run patched the current seed checks with markers such as `MOR_ABADAPPLE_O1`, `MOR_ABADAPPLE_O2`, and `MOR_ABADAPPLE_VIC`.

Validation result: an in-game objective marker was confirmed working by playtest. The launcher now only patches still-locked checks, and the active watcher ignores markers already seen during the current process. Restarting a mission should therefore not grant the same objective reward twice.

Victory-exit result: changing `[Basic] EndOfGame` is not a safe way to stop campaign continuation because it can complete a mission immediately. The launcher now leaves all mission ending fields intact, inserts its marker before the existing terminal win action, grants the victory check from the log signal, and then closes the spawned process tree after a short delay. A static audit found a recognized victory action in all 97 extracted campaign maps.

Reward application result: already-earned reward rules are now merged into the generated map before launch. This replaces the unsafe loose `rulesmo.ini` approach for spawned campaign missions.

Randomizer tech control result: generated maps now force every unit controlled by the reward pool to locked `TechLevel=11`, then apply earned unlocks over that. This prevents campaign maps from handing out randomizer units before the player has earned them.

Immediate reward result: for unlock rewards represented by `TechLevel`, the patched objective/victory trigger also gets a `Set Tech Level for TechType` action (`106`). This should make a newly unlocked unit available during the same mission. Non-tech stat changes still apply at map load only unless a runtime rules-edit action/hook is found.

Loose maps in `MapsMO` show the trigger format we need:

- `[Triggers]` defines named trigger records.
- `[Events]` defines what fires the trigger.
- `[Actions]` defines what happens after the trigger fires.
- Trigger references are also attached to objects in map sections.

Example patterns seen in loose maps:

```ini
[Events]
01000041=1,13,0,300
01000049=1,60,2,1,CAMISC0X

[Actions]
01000041=5,11,4,MISSION:TIMEKILLER01,0,0,0,0,A,19,7,WarningAlarm,0,0,0,0,A,...

[Triggers]
01000041=UnitedStates,<none>,[CHRONO]/Start,0,1,1,1,0
```

## Why This Matters

To unlock items from in-mission objectives, the randomizer needs a reliable way to add or observe trigger completions. There are two likely paths:

1. Extract/copy campaign `.MAP` files to loose files, patch their `[Triggers]`, `[Events]`, and `[Actions]`, and launch those loose maps.
2. Prove that `spawnmap.ini` can inject trigger/action/event sections into an archived campaign scenario.
3. If `spawnmap.ini` is only an output/cache, use it as an analyzer source to identify mission objective triggers, then patch a generated loose copy of the map.

The normal "mission objective complete" UI message is not enough by itself. It appears as trigger actions in the map data, but the current logs do not show a durable fired-objective record that the launcher can read after the game exits. The practical hook is: find the trigger that fires `ObjectiveComplete`, then add one more action or companion signal to that same trigger.

## Next Experiments

1. Build mission-specific objective-to-trigger mappings instead of pairing briefing objectives with raw objective-complete actions by position. The counts differ on 58 of 97 maps.
2. Investigate `SROAD` and `EGODSEND`, which do not expose the standard objective-complete action patterns.
3. Runtime-playtest victory auto-close on one `Winner is` map and one `Announce Win`-only map to confirm the 2.5-second delay preserves the desired victory presentation.
4. If a map action cannot create a durable external signal, use a small companion DLL/memory hook only for trigger-marker reporting.

## Caution

Do not patch campaign maps in-place inside MIX archives yet. Use loose generated copies or backups only.
